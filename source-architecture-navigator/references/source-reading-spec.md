# Source Reading Spec

<!-- @类型: 参考规范 -->
<!-- @目的: 将源码阅读从碎片问答规范成可导航、可验证、可复盘的流程 -->
<!-- @场景: 需要完整理解仓库、函数、调用链、数据流、接口流、配置流或架构风险 -->

> **一句话**: 先分类，再读证据；先局部地图，再全局合并；先边界卡，再施工。
> **版本**: v1.0.0

## 1. 阅读模式

### @步骤1: 默认读模式

<!-- @类型: 规则步骤 -->
<!-- @优先级: 必须 -->
<!-- @验证点: 在用户没有明确要求修改前，不改变仓库内容 -->
<!-- @验证方式: 没有新增调试打印、临时文件或无关格式化 -->
<!-- @ID: spec-read-mode -->

- @动作: 使用检索、只读命令和源码引用建立理解；用户提供 zip 时只解到临时目录，不解压到目标仓库内。
- @动作: 不写临时日志、不插入打印、不保存分析碎片到目标仓库。
- @动作: 若确实需要落盘分析产物，先说明位置和用途，并优先放在仓库外或用户指定目录。

### @步骤2: 施工模式

<!-- @类型: 规则步骤 -->
<!-- @优先级: 必须 -->
<!-- @验证点: 修改前已有施工边界卡 -->
<!-- @验证方式: 能看到目标、涉及文件、不涉及文件、风险、验收、回滚 -->
<!-- @ID: spec-change-mode -->

- @动作: 只有用户明确要求改代码、修 bug、实现需求或重构时，才进入施工模式。
- @动作: 施工前先确认修改边界；施工后用测试、静态检查或最小人工验收验证。
- @动作: 阅读发现的问题不自动变成修改任务。

## 2. 问题分类 spec

| 类别 | 回答重点 | 不要做 |
|---|---|---|
| 理解类 | 职责、输入输出、副作用、局部上下文 | 不直接扩展到全局大图 |
| 定位类 | 入口、定义点、配置点、运行路径 | 不假设唯一答案 |
| 调用链类 | 上游、当前节点、下游、终点 | 不只解释单个文件 |
| 架构类 | 分层、核心路径、边缘路径、模块边界 | 不把所有文件塞进一张图 |
| bug 类 | 证据、复现线索、最小定位路径 | 不先改代码遮盖问题 |
| 需求类 | 目标、影响面、可选方案 | 不马上施工 |
| 重构类 | 证据、收益、风险、最小步 | 不以审美替代证据 |
| 施工类 | 边界卡、实现、验证、回滚 | 不扩大范围 |
| 复盘类 | 问题树、已知、未知、下一步 | 不重复长篇解释 |

## 3. 地图层级 spec

### L0 项目识别卡

最小字段：

- 项目目标：这个系统要完成什么。
- 入口候选：从哪里启动、页面/接口/任务从哪里进入。
- 核心路径：最重要的业务链路。
- 边缘路径：脚本、测试、导出、实验、监控等暂时可后看的路径。
- 配置来源：环境变量、配置文件、启动参数、默认值。
- 当前不确定项：需要继续查证的地方。

### L1 函数地图

每个函数或方法单独成图。不要一开始把多个函数混成复杂图。

最小字段：

- 签名和位置。
- 一句话职责。
- 输入数据来自哪里。
- 输出数据去哪里。
- 内部步骤。
- 副作用：I/O、网络、数据库、全局状态、缓存、日志、任务。
- 上游调用者和下游依赖。
- 风险点或容易误解的点。

### L2 调用链地图

用于回答"谁调用谁"。

推荐结构：

```mermaid
flowchart LR
  A["入口/上游"] --> B["当前对象"]
  B --> C["核心依赖"]
  C --> D["终点/输出"]
```

每条边都应能被搜索结果、import、路由注册、事件绑定或运行入口解释。

### L3 功能流地图

用于回答"一个功能为什么这样流动"。

常见流：

- 页面流：路由 -> 页面组件 -> 状态 -> 接口 -> 展示。
- 接口流：客户端请求 -> 路由 -> 服务 -> 数据访问 -> 响应。
- 数据流：字段诞生 -> 转换 -> 存储 -> 消费 -> 展示/返回。
- 任务流：触发器 -> 队列/调度 -> worker -> 副作用 -> 结果。
- 配置流：默认值 -> 文件 -> 环境变量 -> 启动参数 -> 使用点。

### L4 全局分层图

只有当 L1-L3 已经能解释关键路径后再画。

推荐分层：

- 前端页面层
- 组件层
- 状态层
- 接口层
- 后端路由层
- 业务服务层
- 数据库层
- 任务调度层
- 配置层
- 工具层

规则：

- 全局图必须分层，不做一张无边界的大网。
- 节点数量优先控制在 7-14 个。
- 每个层级给核心文件，不列全量文件。
- 明确"先看这些，暂时不看那些"。

## 4. 证据规则

### @步骤1: 证据等级

<!-- @类型: 规则步骤 -->
<!-- @优先级: 必须 -->
<!-- @验证点: 每个判断都能落到证据等级 -->
<!-- @验证方式: 结论旁边有文件、符号、搜索结果或运行输出 -->
<!-- @ID: spec-evidence-levels -->

| 等级 | 证据类型 | 可用于 |
|---|---|---|
| E1 | 文件路径和符号定义 | 定位、职责判断 |
| E2 | 调用者/被调用者搜索结果 | 调用链判断 |
| E3 | 路由、配置、注册表、manifest | 入口和契约判断 |
| E4 | 测试、运行输出、日志、失败栈 | bug 和行为判断 |
| E5 | 多处重复实现或配置多源 | 结构风险判断 |

每张图和每条阅读路线都要补证据表：

```markdown
| 对象/边/路线项 | 证据等级 | 证据位置 | 状态 |
|---|---|---|---|
| 核心入口 | E3 | `zip内相对路径或仓库相对路径:line` | 已证实 / 推断 |
```

@警告: 若没有 `path:line`，必须标为"推断"，并给出下一步查证方式。

### @步骤2: 风险陈述格式

<!-- @类型: 规则步骤 -->
<!-- @优先级: 必须 -->
<!-- @验证点: 问题不是空泛评价 -->
<!-- @验证方式: 每条发现都有证据、影响、最小下一步 -->
<!-- @ID: spec-risk-format -->

每条架构问题按下面格式输出：

```markdown
[严重度] 问题名
证据:
影响:
为什么现在重要:
最小下一步:
验收方式:
回滚思路:
```

## 5. 特定问题 spec

### 废弃代码判断

不能只靠"看起来没人用"。至少满足两类证据再给结论：

- 无引用或入口不可达。
- 配置/路由/注册表未接入。
- 测试不覆盖且无运行路径。
- 有替代实现且主路径使用替代实现。
- 文件名、注释或版本历史说明过时。

证据不足时写"疑似废弃"，并给最小验证方式。

### 重复代码判断

重复分三类：

- 结构重复：函数骨架类似。
- 语义重复：做同一件事但命名不同。
- 配置重复：默认值或契约分散在多处。

不要直接要求抽象；先判断是否会导致修改不一致。

### 命名混乱判断

优先检查：

- 同一概念在页面、接口、服务、数据库中名字不同。
- 文件名和主要职责不匹配。
- 类型名、函数名和返回结构不匹配。
- "temp/new/old/v2/final" 等无法表达边界的名称。

### 架构断点判断

架构断点是用户理解容易断裂的位置，常见于：

- 类型定义和消费层互相依赖。
- 入口层同时承担业务转换和 I/O。
- 配置默认值分散。
- 数据字段在多个层级被隐式改名。
- 同一功能有两条相似但不一致的路径。

## 6. zip 输入 spec

用户给源码 zip 时：

- 先说明当前仍是读模式，不修改目标仓库。
- 默认解压到会自动清理的系统临时目录；若需要沿临时文件继续深读，使用 `--keep-temp` 或 `--extract-to <仓库外目录>`。
- 检查 zip 成员路径，拒绝绝对路径或包含 `..` 的成员。
- 先检查成员数量、总解压大小、单文件大小；超过限制就停止。
- 只解压源码、文本、manifest 和配置文件，跳过 `__pycache__`、构建产物和二进制生成物。
- 读取完成后，只把证据路径写成 zip 内相对路径或临时目录相对路径。
- 不把临时解包目录作为后续施工目标；真正施工前必须重新确认目标仓库路径。

## 7. 阅读路线 spec

每次给路线只给 3-5 个文件。格式：

```markdown
1. path/to/entry.ext
   读什么:
   解决什么疑问:
   暂时跳过:
   证据: E1/E2/E3 `path/to/entry.ext:line`
2. path/to/core.ext
   读什么:
   解决什么疑问:
   暂时跳过:
   证据: E1/E2/E3 `path/to/core.ext:line`
```

路线排序：

1. 入口和装配。
2. 核心数据结构或状态。
3. 核心转换函数。
4. 边界适配层。
5. 测试或验证入口。

## 8. 一次全解析 HTML 报告 spec

当用户要阅读一个仓库或源码 zip，且任务目标是建立源码架构理解时，优先生成一份静态 HTML 阅读报告。

最小内容：

- L0 项目识别卡：项目目标线索、入口候选、manifest、主要目录、不确定项。
- 解析覆盖矩阵：项目识别、入口、全局分层、模块依赖、逐函数地图、配置、测试、风险候选。
- L3 全局分层架构：启动/脚本层、页面/接口/任务入口层、编排层、核心领域模块、支撑/配置/评估层、测试/验证层。
- L2 模块关系图：模块职责、依赖谁、被谁依赖、耦合度。
- L1 逐函数地图：按模块列出函数/类/方法，给出位置、签名、所属模块职责、输入/输出/副作用线索；禁止只写"可下钻，待确认职责"。
- Golden Path：从入口到核心输出的主路径，不把边缘脚本当主链路。
- 配置依赖与接口契约检查：manifest/config、Input/Output/Config/State/Result 等契约对象候选。
- 问题诊断：入口过多、脚本占比高、测试缺失、耦合偏高等静态可证候选。
- 阅读路线：新手路线和高手路线，两者分开。
- 证据表：每个判断落到 E1-E5，证据不足标为推断。
- 下一问：下钻、横向比较、施工前边界三类。

交互要求：

- 函数/类卡片支持筛选。
- 模块函数组支持展开/收起。
- 主路径可被按钮高亮。
- 证据表可显隐。
- 页面必须有阅读笔记能力：点击任意正文位置写笔记，不显示提示文字、位置文字、清空按钮或多余说明。
- 点击正文位置后，独立笔记层在该锚点附近出现输入点；笔记层和正文内容层隔离，不改写正文 DOM、不挤压原网页布局，也不使用弹窗；点击其它正文位置时自动退出上一个编辑态并切到新位置；无输入则恢复普通状态，有输入则自动保存并变成凸起笔记点。
- 已保存笔记必须可以拖动，拖动后的偏移写入浏览器本地存储；刷新后根据稳定正文锚点和偏移恢复位置，不依赖随机锚点。
- 笔记吸附关系只画在独立笔记层：悬停、编辑或拖动时显示锚点、细连接线和选中文本高亮；正文内容层不插入笔记控件、不被挤压，也不为吸附效果改写 DOM。
- 笔记控件必须是单 HTML 内联实现，依赖浏览器本地存储；滚动条、下拉框和夜间模式也要重绘为报告同款纸张、粗边和低对比凹凸样式。笔记点使用 16px 左右的小圆角、同色系渐变和新拟态阴影，编辑态下沉，保存态凸起，避免胶囊形大圆角。
- 笔记导出必须有两种格式：`按照顺序：<用户记录内容>` 用于按阅读路线复盘；`原文：<选中文本或位置>\n记录：<用户记录内容>` 用于回到具体句子、判断或源码证据。
- 分区进入视口时可以有轻量动画，但不能牺牲可读性。

落盘规则：

- 默认输出到被扫描仓库外，例如用户指定的分析目录。
- 若输入是 zip 且未指定 `--keep-temp` 或 `--extract-to`，HTML 中只保留 zip 内相对证据路径，不承诺可点击文件链接。
- 只有用户明确要求把报告放进仓库时，才使用 `--allow-output-in-repo`。

推荐命令：

```bash
python scripts/render_navigation_html.py <repo-or-source.zip> --output <outside-analysis-dir>/source_navigation_report.html --title "<project-name>"
```

## 9. 施工边界 spec

当用户从阅读转向改代码，使用最小边界卡：

```markdown
目标:
涉及文件:
不涉及文件:
修改边界:
主要风险:
验收方式:
回滚方式:
```

边界卡通过后再施工。若用户明确要求立即实现，可以把边界卡压缩到 5-8 行，但不能省略。
